# OpenSearch

QoreDB embarque un driver OpenSearch natif qui dialogue avec le cluster en HTTP (port `9200`) ou HTTPS. OpenSearch est le fork sous licence Apache 2.0 d'Elasticsearch, et QoreDB le traite comme un moteur de premier plan : explorateur de schéma, grille de résultats, exports et notebooks fonctionnent de la même manière, avec un **éditeur de recherche** pour la Query DSL et un **mode SQL** pour des requêtes de type relationnel.

## Connexion rapide

Choisissez le moteur **OpenSearch** et saisissez l'adresse du cluster. Le port par défaut est `9200`.

```
http://USER:PASSWORD@HOST:9200
https://USER:PASSWORD@HOST:9200
```

Les services OpenSearch managés (y compris AWS OpenSearch Service) sont exposés en HTTPS. Collez le point d'accès et choisissez le mode d'authentification correspondant à votre déploiement.

## Champs de connexion

| Champ           | Requis   | Notes                                                                            |
| --------------- | -------- | -------------------------------------------------------------------------------- |
| **Hôte**        | Oui      | Nom d'hôte ou IP. Point d'accès du domaine pour les services managés.            |
| **Port**        | Oui      | `9200` par défaut.                                                               |
| **TLS**         | Optionnel| À activer pour HTTPS. S'appuie sur Rustls ; aucun OpenSSL système requis.        |
| **Certif. CA**  | Optionnel| Certificat CA personnalisé pour les clusters à autorité privée ou auto-signée.   |

## Modes d'authentification

Le driver prend en charge plusieurs modes d'authentification, sélectionnables dans la fenêtre de connexion :

- **Aucun** — pour un cluster local démarré sans le plugin de sécurité.
- **Basic auth** — nom d'utilisateur et mot de passe (chiffrés dans le vault).
- **Clé API** — envoyée via l'en-tête `Authorization`.
- **Jeton Bearer** — un jeton bearer pour l'authentification par token ou par compte de service.

## Éditeur de recherche et mode SQL

OpenSearch propose deux surfaces de requête dans QoreDB :

- **Éditeur de recherche** — écrivez de la Query DSL sur un index (requêtes `_search`), les résultats s'affichent dans la grille. Idéal pour la recherche plein texte et les agrégations.
- **Mode SQL** — exécutez des instructions `SELECT` via le plugin SQL d'OpenSearch. Pratique pour l'exploration tabulaire et pour réutiliser SQL d'un moteur à l'autre.

Les grands jeux de résultats sont streamés dans la grille au fur et à mesure plutôt que mis en tampon en entier : une requête large commence à afficher des lignes immédiatement.

## Sûreté des requêtes

Les opérations de lecture (recherches, `SELECT` SQL, `_cat` et introspection d'index) s'exécutent librement. Les opérations qui modifient ou administrent le cluster — création/suppression d'index, écritures de documents, `_reindex`, paramètres du cluster — passent par le filet de sécurité universel de QoreDB : le mode lecture seule et les garde-fous d'environnement (dev/staging/prod) s'appliquent exactement comme sur PostgreSQL ou MySQL.

## Problèmes courants

<Callout type="warning">
**Échec du handshake TLS** : les clusters managés et sécurisés exigent HTTPS. Vérifiez que l'URL commence par `https://`, et ajoutez le certificat CA de votre cluster s'il utilise une autorité privée ou auto-signée.
</Callout>

<Callout type="info">
**AWS OpenSearch Service** : les domaines utilisant la signature de requêtes SigV4 plutôt que la basic auth ou des jetons peuvent nécessiter un proxy en amont ; utilisez l'un des modes d'authentification pris en charge pour les connexions directes.
</Callout>
